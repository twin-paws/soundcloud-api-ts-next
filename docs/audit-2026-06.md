# Library Audit — June 2026

**Scope:** full doc/code consistency sweep + code audit of `soundcloud-api-ts-next` (React hooks + Next.js route handlers over `soundcloud-api-ts`), plus the upgrade to `soundcloud-api-ts@^1.14.0` and absorption of its behavior changes.
**Trigger:** the upstream `soundcloud-api-ts` audit (`D:\dev\soundcloud-api-ts\docs\audit-2026-06.md`, 2026-06-10) found *systemic* doc/code drift — flagship config options (`dedupe`, `cache`, `cacheTtlMs`, `fetch` injection, a never-existent `AbortController`) were documented for years while the code silently ignored them. This audit assumes the same disease is present here and verifies every behavioral claim against `src/`.
**Method:** code is the source of truth. Every claim in `README.md`, `AGENTS.md`, `llms.txt`, `llms-full.txt`, `CHANGELOG.md`, `docs/*`, and JSDoc was checked against `src/`. Anything not verifiable from code alone (live SoundCloud endpoint behavior, Next.js runtime behavior) is marked **ASSUMED**.
**Status:** _fixes applied and released as **v1.13.0** (2026-06-10)._ Baseline suite at `soundcloud-api-ts@^1.13.2`: **350 passed / 0 skipped**, 14 files, on Node 24 / pnpm 10.23.0. After the fixes: **351 passed / 0 skipped** (the +1 is the new H1 constructor-passthrough test); build + typecheck + lint re-run green before every fix commit. Commit hashes are in the fix-commits table below.

> **Note on scope:** the security-boundary review (client/server bundle separation, secret leakage, CSRF, cookie flags) is tracked separately and is intentionally **not** included in this report.

## Executive summary

The disease is present, and it is the same one upstream had: **docs describe configuration the code does not implement.** The worst instance is the headline of every agent-facing doc — `AGENTS.md`, `llms.txt`, and `llms-full.txt` all state that `createSoundCloudRoutes(config)` forwards `fetch`, `AbortController`, `dedupe`, `cache`, `cacheTtlMs`, and `onRetry` "through to the underlying `SoundCloudClient`." None of these fields exist on the `SoundCloudRoutesConfig` type, and `getClient()` forwards only `onRequest`. A consumer following the docs writes code that either fails to typecheck (`AbortController`, which never existed even upstream) or silently no-ops.

Beyond that, several user-facing examples are simply wrong against the current API: the README's React Server Components example calls `createSoundCloudServerClient` as if it were synchronous and returned a `SoundCloudClient` (it is `async` and returns a `{ client, userToken, token() }` wrapper), the `CookiePkceStore` constructor is documented with an object argument it doesn't accept, a phantom `createSoundCloudRouteHandler` function name appears in two docs, and `CHANGELOG` 1.10.0 claims `useTracks`/`useMeConnections` hooks and batch/connection routes that were never shipped (only the headless `scFetchers` equivalents exist).

On the upgrade itself: the four potentially-breaking 1.14.0 behavior changes were each traced to this package's call sites and **none break it**. The `redirectUri`-required throw only affects `getUserToken`/`refreshUserToken`, which are called exclusively by `SCAuthManager` — and that class always constructs its client with a `redirectUri` (it's a required config field). Every other client construction in this package (`routes.ts`, `server-helpers.ts`, `fetchers.ts`, `server/client.ts`) only ever calls `getClientToken()` (the `client_credentials`/Basic-Auth grant), which does not require `redirectUri`. Default-on GET deduplication is safe because nothing in the package mutates an API response object in place.

## Findings

### High

**H1 — Phantom config options on `createSoundCloudRoutes` (the core drift).**
`AGENTS.md`, `llms.txt`, and `llms-full.txt` document `createSoundCloudRoutes` as forwarding `fetch`, `AbortController`, `dedupe`, `cache`, `cacheTtlMs`, and `onRetry` to the underlying client.
Evidence:
- `SoundCloudRoutesConfig` (`src/types.ts:60-164`) declares **only** `clientId`, `clientSecret`, `redirectUri`, `getToken`, `onRouteComplete`, `onRequest`, `routes`, `cacheHeaders`, `cors`, `csrfProtection`. None of the six documented options are present.
- `getClient()` (`src/server/routes.ts:22-32`) forwards only `clientId`, `clientSecret`, `redirectUri`, `onRequest` — not even `onRetry`.
- Docs asserting otherwise: `AGENTS.md:98-106` ("Fetch injection — pass `fetch`/`AbortController`", "In-flight deduplication — `dedupe: true` (default)", "Cache interface — pass `cache`", "All config options are passed through `createSoundCloudRoutes(config)` → forwarded to the underlying `SoundCloudClient`"); `llms.txt:112-122`; `llms-full.txt:354-368` (declares them as fields of `SoundCloudRoutesConfig`).
This is exactly upstream's C1/C2/H2 class. `AbortController` is vapor (it never existed on `SoundCloudClientConfig`, even after the upstream fix). The other five are real, working options on `soundcloud-api-ts@1.14.0`'s `SoundCloudClientConfig` (`fetch`, `dedupe`, `cache`, `cacheTtlMs`, `onRetry`, plus `maxRetries`/`retryBaseDelay`/`onDebug`).
**Planned fix:** follow upstream's strategy — wire the *real* options (`fetch`, `dedupe`, `cache`, `cacheTtlMs`, `onRetry`, `maxRetries`, `retryBaseDelay`) into `SoundCloudRoutesConfig` and `getClient()` so the published contract is fulfilled (additive, non-breaking → minor bump); **remove** `AbortController` from all docs. Add a test pinning passthrough.

**H2 — README "Server Components (RSC)" example is broken.**
`README.md:208-225` shows:
```ts
const sc = createSoundCloudServerClient({ clientId, clientSecret });   // no await
const track = await sc.tracks.getTrack(Number(params.id), { revalidate: 60, tags: [...] });
```
Actual API (`src/server/client.ts:106-125`): `createSoundCloudServerClient` is `async` and resolves to `{ client, userToken, token() }`. The correct call is `await createSoundCloudServerClient(...)` then `sc.client.tracks.getTrack(id, { token })`. The underlying `getTrack` takes a `TokenOption`, not `{ revalidate, tags }` — those belong to the *server helper* `getTrack(id, config, cacheOptions)`, a different function. `docs/rsc-guide.md:33-44` shows the correct `await` + `sc.client.*` form, so the README contradicts the guide.
**Planned fix:** rewrite the README RSC snippet to match `client.ts` + `rsc-guide.md`.

### Medium

**M1 — `CookiePkceStore` constructor documented with the wrong signature.**
Actual: `constructor(secret: string, cookieName = "sc_pkce")` — positional (`src/auth/stores/cookie-pkce-store.ts:126-129`). Methods: `setCookieHeader(state, ttlMs, secure?): Promise<string>` (`:185`), `getFromRequest(req): Promise<string | undefined>` (`:240`).
Docs claiming an object argument / wrong return types:
- `AGENTS.md:112` — "Accepts `{ secret: string, cookieName? }`".
- `llms.txt:115` — `new CookiePkceStore({ secret: process.env.COOKIE_SECRET })`.
- `llms.txt:134-136` — `CookiePkceStore({ secret, cookieName? })`, `.setCookieHeader(state, verifier): string`, `.getFromRequest(req): Map<string, string>`.
`README.md:81` and the example in the class JSDoc use the correct positional form. Following the AGENTS/llms form throws at construction.
**Planned fix:** correct `AGENTS.md` and both llms files to positional args and the real method signatures.

**M2 — Phantom function name `createSoundCloudRouteHandler`.**
The exported factory is `createSoundCloudRoutes` (`src/server/routes.ts:517`). The name `createSoundCloudRouteHandler` appears in `AGENTS.md:149` and `CHANGELOG.md:30` and has never existed.
**Planned fix:** rename to `createSoundCloudRoutes` in both.

**M3 — `createSoundCloudServerClient` documented as sync, returning a `SoundCloudClient`.**
`AGENTS.md:122` ("returns a configured `SoundCloudClient`") and `llms-full.txt:139` (`createSoundCloudServerClient({...}): SoundCloudClient`) are both wrong: it is `async` and returns the `SoundCloudServerClient` wrapper (`{ client, userToken, token() }`) (`src/server/client.ts:49-62, 106`).
**Planned fix:** correct both to `Promise<SoundCloudServerClient>` and note the `.client` / `.token()` shape.

**M4 — `CHANGELOG` 1.10.0 describes features that were never shipped.**
`CHANGELOG.md:55-71` (v1.10.0) claims:
- `useTracks(ids[])` hook — **does not exist** (no such file/export; `src/client/index.ts` has no `useTracks`).
- `useMeConnections()` hook — **does not exist**.
- `GET /api/soundcloud/tracks?ids=...` batch route — **not in** `src/server/routes.ts`.
- `GET /api/soundcloud/me/connections` route — **not in** `src/server/routes.ts`.
What *does* exist for these capabilities is the headless layer: `scFetchers.tracks(ids[])` and `scFetchers.meConnections(token)` (`src/fetchers.ts:130-136, 207-209`) plus `scKeys.tracks`/`scKeys.meConnections` (`src/query-keys.ts:27-28, 48`). Same drift class as upstream's M6 (changelog describing code the tag didn't have).
**Planned fix:** add a Keep-a-Changelog correction blockquote under the 1.10.0 entry (preserve history) clarifying that only the `scFetchers`/`scKeys` forms exist; record the accurate state in the new release entry.

### Low

**L1 — `usePlayer` signature wrong in two docs.**
Actual: `usePlayer(trackId: string | number | undefined)` (`src/client/hooks/usePlayer.ts:33`). `README.md:180` and `llms.txt:44` say `usePlayer(streamUrl)`. `llms-full.txt:71` is correct (`usePlayer(trackId)`).
**Planned fix:** correct README + llms.txt.

**L2 — Stale dependency-version references and llms version headers.**
`AGENTS.md:91-106`, `llms.txt:108-129`, `llms-full.txt:108-115,361,385` all anchor on "soundcloud-api-ts v1.13.0+". `llms.txt:1` and `llms-full.txt:1` carry `v1.12.4`. After the dependency bump and version bump these become stale (the precise drift vector from the original incident).
**Planned fix:** update dependency references to 1.14.x and bump the llms headers with the release.

**L3 — CI/publish/docs workflows pin pnpm out of step with the repo.**
`.github/workflows/{ci,publish,docs}.yml` all use `pnpm/action-setup@v4` with a hardcoded `version: 9`, while the local toolchain is pnpm **10.23.0** and there is **no `packageManager` field** in `package.json` (`pnpm-lock.yaml` is `lockfileVersion: '9.0'`). This is the same footgun-class as the upstream red-CI incident (floating/ mismatched pnpm vs lockfile). It is not currently a hard break — a `9.0` lockfile is readable by both pnpm 9 and 10, and CI runs bare `pnpm install` (not `--frozen-lockfile`), so it resolves rather than failing — but CI executing a different pnpm major than every developer is latent drift.
Publish trigger: `publish.yml` fires on `release: published` (GitHub Release trigger, same as upstream) and runs `pnpm install` → `pnpm build` → `npm publish --provenance`. It will inherit the same pnpm-version skew.
**Planned fix:** add `"packageManager": "pnpm@10.23.0"` to `package.json` and drop the hardcoded `version:` from all three workflows so `action-setup` reads `packageManager`. Verify publish workflow green before any release.

**L4 — CI does not exercise Node 20 although `engines` claims `>=20`.**
`package.json:96` `engines.node >= 20`; README badge says "Node ≥20" (consistent); CI matrix tests Node 24 only. Same remaining item as upstream M5/Remaining-#4 — listed, not fixed (adding a Node 20 job or bumping `engines` are both out of scope for a non-breaking pass).

## Task 1 — `soundcloud-api-ts@^1.14.0` behavior changes, traced to this package

Each of the four potentially-breaking changes was checked against every `SoundCloudClient` construction and call site in `src/`. Result: **no code change required for correctness**; the upgrade is safe. (The bump is still a *minor* because newly-activated behavior is exposed once H1 is wired.)

| 1.14.0 change | Effect here | Verdict |
|---|---|---|
| **(a)** GET dedupe active by default (shared promise + **same object reference**) | The only long-lived client is the `routes.ts` module singleton. It `JSON.stringify`s every result immediately and never mutates response objects; `useInfinite` accumulates with `[...prev, ...collection]` (new arrays); browser-side fetches go through Next routes, a separate layer. A repo-wide grep for in-place mutation (`.push`/`.sort`/`.splice`/`json[...] =`/`delete json`) found only `cookie flags.push(...)` and a test array. | **Safe** |
| **(b)** `cache`/`cacheTtlMs` now functional | This package neither forwards them (see H1) nor implements a workaround that depends on them. The client-side dedupe in `src/client/hooks/_useSCFetch.ts` (`inflightMap`) operates at the **browser→Next-route fetch** layer, independent of the upstream `SoundCloudClient`, so it is *not* the no-op workaround upstream described — **keep it.** | **No action** (wiring tracked in H1) |
| **(c)** `config.fetch` reaches all methods | n/a until H1 wires `fetch` through. | **No action** (H1) |
| **(d)** `getUserToken`/`refreshUserToken` **throw** without `redirectUri` | Only `SCAuthManager.exchangeCode`/`refreshToken` call these (`src/server/auth.ts:257,268`), and `SCAuthManager` always constructs its client with `redirectUri` (required field; `auth.ts:198-202`). `routes.ts`, `server-helpers.ts`, `fetchers.ts`, `server/client.ts` construct clients without `redirectUri` but only ever call `getClientToken()` (Basic-Auth `client_credentials`, no `redirectUri` — confirmed `SoundCloudClient.ts:345-359` upstream). | **Safe** |
| **(e)** `sc.auth.*` honors retry options | No behavioral reliance here; would only matter once retry options are forwarded (H1, for the non-auth client). | **No action** |
| **(f)** telemetry (`onRequest`) does **not** fire for `sc.raw.*` / `auth.signOut` | `routes.ts` uses standalone `signOut(token)` (`:192`) — no telemetry, consistent with the exclusion. Separately, the `/next` pagination route calls `scFetchUrl(nextUrl, token)` **without** passing `onRequest` (`routes.ts:358-360`), so paginated continuation fetches are not telemetered — matches the upstream "pagination not deduped/telemetered" caveat. | **No action** (minor doc note candidate) |

## Code audit (non-security)

Verified correct, no action:
- **Hook deps / stale closures:** `_useSCFetch` keeps `transform` in a ref and excludes it from the effect deps with an explicit eslint-disable + rationale (`_useSCFetch.ts:119,182-184`); the effect deps `[urlOrNull, enabled, retry, refreshInterval, authHeader]` are complete for what it reads. `useInfinite` memoizes `fetchPage` on `[apiPrefix]` and guards unmount with `mountedRef`/`abortRef` (`useInfinite.ts:40-91`). Infinite wrappers (`useInfiniteUserTracks` etc.) `useMemo` the URL on the id. Action hooks (`useFollow`/`useLike`/`useRepost`) `useCallback` on `[apiPrefix, accessToken]`.
- **SSR safety:** all hook files and `provider.tsx` are `"use client"`; no top-level `window`/`document` access (the `usePlayer` `new Audio(...)` is inside an effect, `usePlayer.ts:58`).
- **Error propagation:** the route handler maps a thrown upstream error's `statusCode` to the response status and emits the `{ code, message, status, requestId }` envelope (`routes.ts:632-648`); `SoundCloudError` carries `status` + `is401/is403/is404/is429` getters upstream (`errors.ts:95-128`). Consumers branching on `err.status` for 401/403/429 get the status through the envelope. **ASSUMED:** exact live status mapping for SoundCloud 5xx vs 4xx (not executed against the live API).
- **Token-refresh flow:** the per-grant OAuth wire formats are handled **upstream** and not re-implemented here — this package has no direct `/oauth/token` fetch; `SCAuthManager` delegates to `sc.auth.getUserToken`/`refreshUserToken` (`auth.ts:257,268`), and the credential-token paths delegate to `sc.auth.getClientToken()`. Compliant with the "must not re-implement wire formats" rule.
